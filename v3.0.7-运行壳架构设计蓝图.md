## 版本说明

本文档基于 v3.0 适配器插件架构演进方案，提出更高层次的抽象——**运行壳（Runtime Shell）** 概念，将 Port/Adapter 模式提升为**能力契约模型**，实现插件与基础设施的彻底解耦。

### v3.0 核心目标
1. **运行壳能力模型** - 建立 Runtime Capability Contract，屏蔽运行环境差异
2. **插件完全解耦** - 插件仅依赖能力契约，不依赖任何基础设施
3. **双模式商业支撑** - 支持独立部署模式（Standalone）与嵌入模式（Embedded）
4. **前后端完全分离** - 前后端采用统一的 Runtime + Plugin 思想，且前后端代码彻底分离

### v3.0.7 更新内容（2026-03-02）
- **架构重构**：层次命名专业化 — Layer 2 拆分为 2A/2B/2C 三个子领域

  - **Layer 2B**：共享运行时层（Runtime） — 运行时依赖提供（仅前端需要）
  - **Layer 2C**：能力实现层（Implementation） — 契约的具体实现

## 1. 核心概念定义

### 1.1 概念术语表

| 术语 | 定义 | 说明 |
|------|------|------|
| **绝端部署模式（Standalone Mode）** | 产品整体售卖 | 客户购买完整 Brix Platform |
| **嵌入模式（Embedded Mode）** | 单独售卖插件组合的 Service | 插件作为独立产品嵌入客户系统 |
| **运行壳（Runtime Shell）** | 平台为插件提供的运行时能力契约 | 屏蔽运行环境差异、基础设施实现和宿主形态 |
| **能力契约（Capability Contract）** | Runtime Shell 提供的抽象接口 | 插件唯一允许依赖的接口层 |
| **Host（宿主）** | **极薄组装壳**，纯配置驱动的能力编排层 | **极薄设计**：零实现代码、零业务逻辑、零工具类，仅含启动类 + 依赖声明 + YAML 配置 |
| **Adapter（适配器）** | 基础设施能力的具体实现 | 如 Kafka EventBus、Redis StateStore |
| **Shared Runtime（共享运行时）** | **前端运行时依赖的唯一真相源** | 提供 React、Router、State、UI 等运行时库，解决 Module Federation 多实例问题 |
| **Platform Commons（平台通用）** | 平台级通用能力实现 | 鉴权、网关、可观测性等 |
| **Plugin（插件）** | 最小可售卖、可组合的业务单元 | 仅依赖 Capability Contract |
| **Domain Event（领域事件）** | 插件内部的业务事件 | 业务语义，插件内传递 |
| **Integration Event（集成事件）** | 跨插件/跨系统的集成事件 | 集成契约，由 Runtime Shell 路由 |

### 1.2 核心设计约束

```text
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                              v3.0 运行壳核心设计约束                                     │
├─────────────────────────────────────────────────────────────────────────────────────────┤
│                                                                                          │
│  约束1：Runtime Shell 是能力模型，不是具体系统                                           │
│  ═══════════════════════════════════════════════════════════════════════════════════════│
│  • Runtime Shell 是一组接口契约（Capability Contract）                                  │
│  • 不是框架、不是中间件、不是独立进程                                                    │
│  • 由不同 Host 组装具体实现（Host 本身不含实现代码）                                     │
│                                                                                          │
│  约束2：插件只依赖 Runtime Capability Contract                                          │
│  ═══════════════════════════════════════════════════════════════════════════════════════│
│  • 插件代码中禁止出现 Kafka / Redis / HTTP Client 等基础设施依赖                        │
│  • 插件不得绕过 Runtime Shell 直接访问基础设施                                          │
│  • Runtime Shell 是唯一允许接触基础设施的层                                             │
│                                                                                          │
│  约束3：Host 能力等价原则                                                               │
│  ═══════════════════════════════════════════════════════════════════════════════════════│
│  • Standalone Host 与 Embedded Host 提供相同的 Capability 接口                          │
│  • 区别仅在实现复杂度，不在功能可达性                                                    │
│  • 不允许为嵌入模式设计"特例架构"                                                       │
│                                                                                          │
│  约束4：插件间通过事件解耦                                                              │
│  ═══════════════════════════════════════════════════════════════════════════════════════│
│  • 插件之间互不感知，只通过事件通信                                                      │
│  • 插件声明"我能发布什么 / 我能订阅什么"，不写具体监听逻辑                               │
│  • Runtime Shell 负责事件路由与转换                                                     │
│                                                                                          │
│  约束5：运行壳复杂度对插件开发者不可见                                                   │
│  ═══════════════════════════════════════════════════════════════════════════════════════│
│  • 插件开发者只需了解 Capability Contract                                               │
│  • 事件如何路由、数据如何持久化，由 Runtime Shell 处理                                   │
│  • 所有跨插件通信必须可观测                                                             │
│                                                                                          │
│  约束6：Host 极薄原则（Ultra-Thin Host）⭐ New                                          │
│  ═══════════════════════════════════════════════════════════════════════════════════════│
│  • Host 是纯配置驱动的组装壳，不含任何功能实现代码                                      │
│  • Host 项目仅包含：Spring Boot 启动类 + 依赖声明（pom.xml）+ YAML 配置                 │
│  • Host 内禁止出现：Service 类、工具类、Helper 类、自定义 Bean 定义                      │
│  • Host 内禁止出现：if/else 逻辑、循环逻辑、数据转换逻辑                                │
│  • Host 的代码量度量：Java 代码 ≤ 1 个启动类（< 30 行），其余全是配置文件               │
│  • 所有能力实现必须来自 Layer 2C（infra-adapters / platform-commons）的依赖             │
│  • 能力切换通过 Spring Profile / YAML 配置完成，绝不通过代码分支                        │
│                                                                                          │
│  【违反约束的后果】                                                                      │
│  • 约束违反：插件无法在不同 Host 间移植                                                 │
│  • 约束违反：版本分裂、维护成本指数级增长                                                │
│  • 约束违反：架构腐化、平台治理失控                                                      │
│                                                                                          │
│  约束7：前后端完全分离原则（Full-Stack Separation）⭐ New                                │
│  ═══════════════════════════════════════════════════════════════════════════════════════│
│  • 前端 UI 层禁止混入后端调用逻辑，所有数据获取必须经 Repository → HttpCapability 链路   │
│  • 前端采用三层分离架构：View（pages/components）→ ViewModel（hooks）→ Repository        │
│  • View 层为纯渲染层，禁止直接调用 HTTP / fetch / axios                                 │
│  • ViewModel 层（hooks）负责状态管理与业务编排，通过 Repository 获取数据                 │
│  • Repository 层通过工厂模式注入 HttpCapability，封装全部后端 API 调用                   │
│  • {name}-shared 模块作为前后端类型契约，包含 DTO 类型、事件定义、权限常量               │
│  • 大规模修改 UI 样式/交互时，后端代码零改动；后端 API 变更时，仅需修改 Repository 层    │
│                                                                                          │
│  约束8：前端运行时唯一真相源原则（Shared Runtime Single Source of Truth）⭐ New           │
│  ═══════════════════════════════════════════════════════════════════════════════════════│
│  【背景】Module Federation 运行时动态加载模式下，多插件可能各自打包 React，导致：         │
│    - 浏览器中同时存在多个 React 实例，违反 Hooks 规则，应用崩溃                        │
│    - 版本碎片化（React ^18.0.0 ~ ^18.3.1、react-router ^6.0.0 ~ ^6.28.0）                │
│    - Context 断裂，插件间无法共享状态                                                   │
│                                                                                          │
│  【约束内容】                                                                            │
│  • 所有前端运行时依赖（React、Router、State、UI）必须从 @brix/shared-runtime-web 获取   │
│  • 禁止 infra-adapters、platform-commons、plugins 直接在 dependencies 中声明 React 等   │
│  • 运行时依赖必须统一使用 peerDependencies 指向 @brix/shared-runtime-web               │
│  • shared-runtime 提供 MF shared 配置，禁止各模块手写 eager: true                       │
│  • 版本由 shared-runtime 统一定义，消除版本碎片化                                       │
│                                                                                          │
│  【与后端对比】                                                                          │
│  • 后端：Maven BOM + platform-parent 在编译时统一版本，不存在运行时共享问题             │
│  • 前端：Module Federation 是运行时动态加载，必须有统一的运行时提供者                   │
│                                                                                          │
└─────────────────────────────────────────────────────────────────────────────────────────┘
```

#### 2.1.2 前端共享运行时层（Shared Runtime Layer）⭐ New

> **问题背景**：Module Federation 运行时动态加载模式下，Host 和各插件可能各自打包 React，  
> 导致浏览器中同时存在多个 React 实例，违反 Hooks 规则，应用崩溃。

**为什么需要 Shared Runtime 层？**

```text
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                              前端 vs 后端 运行时模型对比                                 │
├─────────────────────────────────────────────────────────────────────────────────────────┤
│                                                                                          │
│  【前端 - 浏览器单进程模型】                                                             │
│  ┌────────────────────────────────────────────────────────────────────────────────────┐ │
│  │                          浏览器 Tab (单一 JS 运行时)                                │ │
│  │  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐                              │ │
│  │  │ Host App     │  │ Plugin A     │  │ Plugin B     │                              │ │
│  │  │              │  │              │  │              │                              │ │
│  │  │  React ──────┼──┼──────────────┼──┼──────────────│ ← 必须共享同一实例！          │ │
│  │  │              │  │              │  │              │                              │ │
│  │  └──────────────┘  └──────────────┘  └──────────────┘                              │ │
│  │                                                                                     │ │
│  │  ⚠️ 如果每个模块各自打包 React：                                                    │ │
│  │     → 内存中存在多个 React 实例 → useState 跨实例调用 → Hooks 规则违反 → 崩溃       │ │
│  └────────────────────────────────────────────────────────────────────────────────────┘ │
│                                                                                          │
│  【后端 - 进程隔离或编译时统一】                                                         │
│  ┌────────────────────────────────────────────────────────────────────────────────────┐ │
│  │  Standalone Host（单一 JVM）                                                        │ │
│  │  ┌──────────────────────────────────────────────────────────────────────────────┐  │ │
│  │  │  plugin-booking.jar   plugin-identity.jar   plugin-partners.jar              │  │ │
│  │  │                              │                                                │  │ │
│  │  │                    Maven BOM 编译时已统一版本                                  │  │ │
│  │  │                    打包成单一 JAR，不存在运行时多版本问题                       │  │ │
│  │  └──────────────────────────────────────────────────────────────────────────────┘  │ │
│  │                                                                                     │ │
│  │  ✅ 后端不需要 Shared Runtime 层 —— Maven BOM + platform-parent 已解决              │ │
│  └────────────────────────────────────────────────────────────────────────────────────┘ │
│                                                                                          │
└─────────────────────────────────────────────────────────────────────────────────────────┘
```

**Shared Runtime 层职责**：

| 职责 | 说明 |
|------|------|
| **依赖唯一性** | React, Router, Zustand, MUI 等运行时库由此层统一提供 |
| **版本一致性** | 消除版本碎片化（当前存在 React ^18.0.0 ~ ^18.3.1） |
| **MF 共享配置** | 提供 `getHostSharedConfig()` / `getRemoteSharedConfig()` |
| **全局注入** | 兼容旧代码的 `window.React` / `window.ReactDOM` 注入 |

**依赖关系变化**：

```text
【改造前 - 版本碎片化】                   【改造后 - 统一真相源】
                                          
Host ─────────────┐                       Host ─────────────┐
  └─ react ^18.2.0 │                        └─ @brix/shared-runtime-web
                   │                                    │
Plugin A ─────────┤                       Plugin A ─────┤──► peerDep: shared-runtime
  └─ react ^18.3.1 │ ← 版本冲突！                       │
                   │                       Plugin B ─────┤
Plugin B ─────────┤                                      │
  └─ react ^18.0.0 │                       infra-adapter-router-web
                   │                         └─ peerDep: shared-runtime
infra-adapter ────┘                                      │
  └─ react-router ^6.0.0                   └─────────────┘
                                                    ↓
                                           shared-runtime-web
                                             └─ react ^18.2.0
                                             └─ react-router-dom ^6.22.0
                                             └─ zustand ^4.5.0
                                             └─ @mui/material ^7.0.0
```

**与其他层的区别**：

| 层 | 职责 | 可替换性 |
|----|------|----------|
| **infra-adapters** | 将契约适配到具体实现（如 Kafka → Webhook） | ✅ 可替换 |
| **platform-commons** | 平台通用业务能力（鉴权、i18n、导航） | ✅ 可配置 |
| **shared-runtime** | 运行时依赖本身 | ❌ 不可替换（React 就是 React） |

#### 2.1.3 能力层子领域说明（2A/2B/2C）⭐ New

> **设计思想**：Layer 2（能力层）是同一个逻辑层，但内含三个不同职责的子领域。  
> 采用 2A/2B/2C 命名而非 2.1/2.2/2.3，是因为它们不是层级关系，而是**并列的子领域**。

```text
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                              Layer 2: 能力层（Capability Layer）                         │
│                              ─────────────────────────────────────────                   │
│                                                                                          │
│  ┌─────────────────────────┐  ┌─────────────────────────┐  ┌─────────────────────────┐ │
│  │   Layer 2A: Contract    │  │   Layer 2B: Runtime     │  │ Layer 2C: Implementation │ │
│  │   ──────────────────    │  │   ────────────────      │  │ ─────────────────────── │ │
│  │   能力契约层（WHAT）     │  │   共享运行时层（WITH）   │  │   能力实现层（HOW）      │ │
│  │                         │  │                         │  │                         │ │
│  │   • 纯接口定义          │  │   • React / Router      │  │   • infra-adapters      │ │
│  │   • 零实现依赖          │  │   • Zustand / MUI       │  │   • platform-commons    │ │
│  │   • runtime-sdk-api     │  │   • 版本唯一真相源       │  │   • 适配器 + 平台能力    │ │
│  │                         │  │   • shared-runtime-web  │  │                         │ │
│  │   定义：有什么能力       │  │   提供：用什么工具       │  │   实现：怎么做到         │ │
│  └─────────────────────────┘  └─────────────────────────┘  └─────────────────────────┘ │
│                                                                                          │
│  【为什么用 A/B/C 而不用 2.1/2.2/2.3？】                                                 │
│  • 2.1/2.2/2.3 暗示层级关系（2.1 在 2.2 上面）                                          │
│  • A/B/C 暗示并列关系（三个子领域平等）                                                  │
│  • 遵循国际行业惯例：Zone A / Zone B / Zone C                                           │
│                                                                                          │
│  【各子领域特征】                                                                        │
│  ┌───────────────────────────────────────────────────────────────────────────────────┐ │
│  │  Layer 2A (Contract)                                                              │ │
│  │  • 定位：能力契约，告诉插件"可以用什么"                                            │ │
│  │  • 特点：零运行时依赖，纯 TypeScript 接口                                          │ │
│  │  • 代码：runtime-sdk-api-web / runtime-sdk-react                                  │ │
│  │  • 依赖：被 Plugin 依赖，被 2B 和 2C 实现                                         │ │
│  └───────────────────────────────────────────────────────────────────────────────────┘ │
│  ┌───────────────────────────────────────────────────────────────────────────────────┐ │
│  │  Layer 2B (Runtime) — 仅前端需要                                                  │ │
│  │  • 定位：运行时依赖真相源，解决 Module Federation 多实例问题                       │ │
│  │  • 特点：版本唯一，不可替换（React 就是 React）                                    │ │
│  │  • 代码：@brix/shared-runtime-web                                                 │ │
│  │  • 说明：后端不需要 2B，Maven BOM 在编译时已统一版本                               │ │
│  └───────────────────────────────────────────────────────────────────────────────────┘ │
│  ┌───────────────────────────────────────────────────────────────────────────────────┐ │
│  │  Layer 2C (Implementation)                                                        │ │
│  │  • 定位：能力契约的具体实现                                                       │ │
│  │  • 特点：可替换，可配置（Kafka ↔ Webhook，React ↔ Vue）                          │ │
│  │  • 代码：infra-adapters + platform-commons                                        │ │
│  │  • 依赖：依赖 2A（实现契约）+ 2B（获取运行时）                                     │ │
│  └───────────────────────────────────────────────────────────────────────────────────┘ │
│                                                                                          │
└─────────────────────────────────────────────────────────────────────────────────────────┘
```

### 2.2 层次职责说明

| 层次 | 名称 | 职责 | 可见性 |
|------|------|------|--------|
| **Layer 4** | 售卖层 | 商业包装、SKU 定义、许可管理 | 商务/运营 |
| **Layer 3** | Host 层（极薄组装壳） | **极薄设计**：仅 pom.xml 依赖声明 + YAML 配置 + 启动类，**零实现代码** | 平台团队 |
| **Layer 2C** | 能力实现层（Implementation） | 能力契约的具体实现（Adapter + Platform Commons） | 平台团队 |
| **Layer 2B** | 共享运行时层（Runtime）⭐ New | 前端运行时依赖唯一真相源（React、Router、State、UI），版本统一管理 | 平台团队 |
| **Layer 2A** | 能力契约层（Contract） | 能力契约定义（纯接口，零依赖） | 平台团队 |
| **Layer 1** | 插件层（Plugin） | 业务逻辑实现、事件声明 | 插件开发者 |
| **Layer 0** | 基础设施层 | Kafka/Redis/DB 等基础设施 | 平台团队（对插件不可见） |

### 3.3.1 DatabaseCapability - 数据库能力（新增）⭐

数据库能力是平台的标准能力，支持配置驱动的多数据库厂商切换。

```text

│                    DatabaseCapability - 数据库能力                                       │
├─────────────────────────────────────────────────────────────────────────────────────────┤

│  │       */                                                                       │    │
│  │      DataSource getDataSource();                                               │    │

│  │      /**                                                                       │    │
│  │       * 获取数据库名称（由 Host 根据插件 ID + 租户确定）                         │    │
│  │       * @return 数据库名称                                                      │    │
│  │       */                                                                       │    │

│  │                                                                                │    │

│  │       */                                                                       │    │
│  │      String getSchemaName();                                                   │    │
│  │                                                                                │    │

│  │       * 执行原生 SQL（用于特殊场景）                                            │    │
```

#### Host 实现示例

```java
/**
 * Standalone Host 的数据库能力实现
 */
@Capability(
    contract = DatabaseCapability.class,
    level = CapabilityLevel.STANDARD,
    description = "数据库能力 - 支持 PostgreSQL/Kingbase 多厂商"
)
@Component
@ConditionalOnProperty(name = "brix.database.enabled", havingValue = "true", matchIfMissing = true)
public class DatabaseCapabilityImpl implements DatabaseCapability {
    
    private final DataSource dataSource;
    private final DatabaseDialect dialect;
    private final String databaseName;
    private final String schemaName;
    private final DatabaseProperties properties;
    
    public DatabaseCapabilityImpl(DatabaseProperties properties) {
        this.properties = properties;
        
        // 根据区域选择数据库厂商
        String region = properties.getRegion();
        String vendor = properties.getRegionMapping().get(region);
        
        // 创建对应的 DataSource
        this.dialect = DatabaseDialect.valueOf(vendor.toUpperCase());
        this.dataSource = createDataSource(vendor, properties);
        
        // 构建数据库名
        this.databaseName = buildDatabaseName(properties);
        this.schemaName = buildSchemaName(properties);
    }
    
    private DataSource createDataSource(String vendor, DatabaseProperties props) {
        HikariConfig config = new HikariConfig();
        
        if ("kingbase".equals(vendor)) {
            // Kingbase 配置
            config.setDriverClassName("com.kingbase8.Driver");
            config.setJdbcUrl(String.format("jdbc:kingbase8://%s:%d/%s",
                props.getKingbase().getHost(),
                props.getKingbase().getPort(),
                databaseName));
            config.setUsername(props.getKingbase().getUsername());
            config.setPassword(props.getKingbase().getPassword());
        } else {
            // PostgreSQL 配置
            config.setDriverClassName("org.postgresql.Driver");
            config.setJdbcUrl(String.format("jdbc:postgresql://%s:%d/%s",
                props.getPostgresql().getHost(),
                props.getPostgresql().getPort(),
                databaseName));
            config.setUsername(props.getPostgresql().getUsername());
            config.setPassword(props.getPostgresql().getPassword());
        }
        
        return new HikariDataSource(config);
    }
    
    @Override
    public DataSource getDataSource() {
        return dataSource;
    }
    
    @Override
    public DatabaseDialect getDialect() {
        return dialect;
    }
    
    @Override
    public String getDatabaseName() {
        return databaseName;
    }
    
    @Override
    public String getSchemaName() {
        return schemaName;
    }
}
```

#### 3.4.2 RuntimeContext 接口设计

`RuntimeContext` 作为插件访问能力的唯一入口，采用 **Registry-Based Default Methods** 设计：

```java
public interface RuntimeContext {
    
    // ===== 核心方法：获取能力注册表 =====
    CapabilityRegistry getCapabilityRegistry();
    
    // ===== 便捷方法：通过 default 实现委托给 Registry =====
    default <T> Optional<T> getCapability(Class<T> capabilityType) {
        return getCapabilityRegistry().get(capabilityType);
    }
    
    default <T> T getCapabilityRequired(Class<T> capabilityType) {
        return getCapabilityRegistry().getRequired(capabilityType);
    }
    
    // ===== 核心能力快捷访问（向后兼容） =====
    default EventBusCapability getEventBus() {
        return getCapabilityRequired(EventBusCapability.class);
    }
    
    default StateStoreCapability getStateStore() {
        return getCapabilityRequired(StateStoreCapability.class);
    }
    
    // ... 其他能力的便捷方法 ...
}
```

**设计优势**：
- 插件代码无感知升级：原有 `ctx.getEventBus()` 调用方式保持不变
- 新能力访问：插件可通过 `ctx.getCapability(NewCapability.class)` 获取新能力
- 能力探测：`ctx.getCapabilityRegistry().isAvailable(SomeCapability.class)`

通过 YAML 配置声明能力需求，实现声明式装配：


#### 3.4.5 与传统方式对比

| 维度 | 传统硬编码方式 | Capability Registry 模式 |

| 能力声明 | 构造函数参数硬编码 | 配置文件声明式 |
| 能力发现 | 手动依赖注入 | `@Capability` 注解自动发现 |
| 扩展新能力 | 修改 AutoConfiguration + RuntimeContext | 只需添加 `@Capability` Bean |
| 可选能力处理 | 条件化 Bean + 空实现 | `registry.get()` 返回 Optional |
| 能力验证 | 无（依赖注入失败才报错） | 启动时显式验证 |
| 运行时安全 | 无保护 | `freeze()` 防止篡改 |
| 能力元数据 | 无 | `CapabilityDescriptor` 完整描述 |

```java
public class ReservationService {
    
    // 仅注入 Capability 接口
    private final EventBusCapability eventBus;
    private final StateStoreCapability stateStore;
    private final AuthContextCapability authContext;
    private final ObservabilityCapability observability;
    
    public ReservationService(
            EventBusCapability eventBus,
            StateStoreCapability stateStore,
            AuthContextCapability authContext,
            ObservabilityCapability observability) {
        this.eventBus = eventBus;
        this.stateStore = stateStore;
        this.authContext = authContext;
        this.observability = observability;
    }
    
    public Reservation create(CreateReservationCommand cmd) {
        // 权限检查
        if (!authContext.hasPermission("reservation:create")) {
            throw new UnauthorizedException("No permission to create reservation");
        }
        
        // 业务逻辑
        Reservation reservation = Reservation.create(cmd);
        reservationRepository.save(reservation);
        
        // 缓存
        stateStore.put("reservation:" + reservation.getId(), reservation, Duration.ofHours(1));
        
        // 发布集成事件（不知道会去 Kafka 还是 HTTP）
        eventBus.publishIntegration(new ReservationCreated(
            reservation.getId(),
            reservation.getCustomerId(),
            reservation.getStatus()
        ));
        
        // 日志（不依赖具体日志框架）
        observability.log(LogLevel.INFO, "Reservation created: {}", reservation.getId());
        
        return reservation;
    }
    
    /**
     * 事件处理器 - 在 manifest 中声明订阅关系
     */
    public void onUserCreated(UserCreated event) {
        // 处理来自 plugin-user 的事件
        observability.log(LogLevel.INFO, "User created, initializing reservation profile: {}", 
            event.getUserId());
    }
}
```

## 6. Host 极薄设计规范

> **v3.0.3 极薄设计原则**：Host 层是纯配置驱动的极薄组装壳（Ultra-Thin Host），**零实现代码、零业务逻辑、零工具类**。  
> Host 项目仅包含 **3 类内容**：① Spring Boot 启动类（< 30 行）② pom.xml 依赖声明 ③ YAML 配置文件。  
> 下文所述的各种 Implementation（如 Kafka + Outbox、Redis Cluster 等）均来自 **infra-adapters** 和 **platform-commons** 的 jar 依赖。  
> Host 切换能力的唯一方式是：更换 pom.xml 中的 adapter 依赖 + 修改 YAML 配置，**绝不通过代码分支**。

### 6.0 极薄设计度量指标

| 度量项 | 标准 | 说明 |
|--------|------|------|
| Java 源文件数 | ≤ 1 | 仅允许 1 个 Spring Boot 启动类 |
| Java 代码行数 | < 30 行 | 启动类仅含 `@SpringBootApplication` + `main()` |
| 自定义 @Bean 数 | 0 | 所有 Bean 由 adapter/commons jar 的 auto-config 提供 |
| Service/Helper 类 | 0 | 禁止任何自定义服务类或工具类 |
| if-else 语句 | 0 | 禁止任何条件分支逻辑 |
| 配置文件数 | ≤ 3 | application.yml + profile-specific YAML |

### 6.0.1 Host 极薄设计反模式

```text
┌─────────────────────────────────────────────────────────────────────────────────────┐
```

### 7.5 前端分层代码示例 ⭐ New

#### 7.5.1 Repository 层（数据访问抽象）

```typescript
// booking-ui-web/src/repositories/BookingRepository.ts

import type { HttpCapability } from '@brix/runtime-sdk-api-web';
import type { Booking, BookingStatus, CreateBookingRequest } from 'booking-shared';

const API_BASE = '/api/v1/bookings';

export interface BookingRepository {
  list(params?: BookingQueryParams): Promise<PagedResponse<Booking>>;
  getById(id: number): Promise<Booking>;
  create(data: CreateBookingRequest): Promise<Booking>;
  updateStatus(id: number, status: BookingStatus): Promise<Booking>;
  delete(id: number): Promise<void>;
}

// 工厂模式：注入 HttpCapability，返回 Repository 实现
export function createBookingRepository(http: HttpCapability): BookingRepository {
  return {
    async list(params?) {
      return http.get<PagedResponse<Booking>>(API_BASE, params);
    },
    async getById(id) {
      return http.get<Booking>(`${API_BASE}/${id}`);
    },
    async create(data) {
      return http.post<Booking>(API_BASE, data);
    },
    async updateStatus(id, status) {
      return http.put<Booking>(`${API_BASE}/${id}/status`, { status });
    },
    async delete(id) {
      return http.delete<void>(`${API_BASE}/${id}`);
    },
  };
}
```

#### 7.5.2 ViewModel 层（状态管理 + 业务编排）

```typescript
// booking-ui-web/src/hooks/useBookingList.ts

import { useState, useEffect, useCallback, useMemo } from 'react';
import { useHttp, useEventBus, useAuth } from '@brix/runtime-sdk-react';
import { createBookingRepository } from '../repositories';
import type { Booking, PagedResponse } from 'booking-shared';
import { BOOKING_EVENTS, BOOKING_PERMISSIONS } from 'booking-shared';

export function useBookingList() {
  // 1. 从 RuntimeContext 获取平台能力
  const http = useHttp();
  const eventBus = useEventBus();
  const auth = useAuth();

  // 2. 工厂模式创建 Repository（http 引用稳定时不重建）
  const repository = useMemo(
    () => http ? createBookingRepository(http) : null,
    [http]
  );

  // 3. 本地状态
  const [bookings, setBookings] = useState<Booking[]>([]);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState<string | null>(null);

  // 4. 权限检查（来自 booking-shared 常量）
  const permissions = useMemo(() => ({
    canCreate:  auth.hasPermission(BOOKING_PERMISSIONS.CREATE),
    canConfirm: auth.hasPermission(BOOKING_PERMISSIONS.CONFIRM),
  }), [auth]);

  // 5. 数据获取（通过 Repository，非直接 HTTP）
  const fetchData = useCallback(async () => {
    if (!repository) return;
    setLoading(true);
    try {
      const result = await repository.list();
      setBookings(result.content);
    } catch (e) {
      setError('加载预约列表失败');
    } finally {
      setLoading(false);
    }
  }, [repository]);

  return { bookings, loading, error, permissions, fetchData };
}
```


```text
│  ✗ 禁止：runtime-sdk-api 依赖任何具体实现模块                                           │
│  ✓ 允许：单向依赖链（A → B → C）                                                        │
│  ✓ 允许：通过事件解耦双向通信需求                                                        │
│  ✓ 允许：依赖共享契约模块（{name}-shared）                                              │
│  【来源】Clean Architecture / SOLID 依赖倒置原则                                         │
│                                                                                          │
│  红线 12：安全边界不可绕过（Security Boundary）⭐ New                                     │
│  ═══════════════════════════════════════════════════════════════════════════════════════│
│  ✗ 禁止：插件自行实现认证逻辑（跳过 AuthCapability）                                     │
│  ✗ 禁止：在日志/错误信息中输出敏感数据（密码、Token、身份证号等 PII）                    │
│  ✗ 禁止：硬编码凭证（API Key、数据库密码、Secret）                                       │
│  ✗ 禁止：插件间直接传递用户凭证（Token 由 Runtime Shell 统一注入）                       │
│  ✓ 允许：通过 AuthCapability 获取当前用户上下文                                          │
│  ✓ 允许：通过 ConfigStore 获取加密的敏感配置                                             │
│  【来源】AWS IAM / Zero Trust Architecture                                               │
│                                                                                          │
│  红线 13：跨服务事件一致性（Transactional Outbox）⭐ New                                  │
│  ═══════════════════════════════════════════════════════════════════════════════════════│
│  ✗ 禁止：在业务事务外直接调用 KafkaTemplate.send() / RabbitTemplate.send()              │
│  ✗ 禁止：业务数据写入与事件发送不在同一事务边界内（双写问题）                            │
│  ✗ 禁止：使用 @TransactionalEventListener + @Async 绕过 Outbox（可能丢事件）            │
│  ✗ 禁止：插件代码直接 new KafkaProducer() 发送消息                                       │
│  ✓ 允许：通过 EventBusCapability.publish() 发布事件（自动走 Outbox）                    │
│  ✓ 允许：显式使用 OutboxRepository 写入事件（与业务数据同事务）                          │
│  ✓ 允许：Runtime Shell 的 CDC/轮询组件负责可靠投递                                       │
│                                                                                          │
│  ⚡ 事件可靠性分类（Outbox 不是所有场景都必须）                                           │
│  ┌──────────────────┬─────────────────────────────────┬──────────────────┬─────────────┐│
│  │ 类型             │ 示例                            │ 丢失后果         │ 发送策略    ││
│  ├──────────────────┼─────────────────────────────────┼──────────────────┼─────────────┤│
│  │ 关键事件         │ 订单支付、合同签署、库存扣减    │ 业务不一致/资损  │ 必须 Outbox ││
│  │ 非关键事件       │ 欢迎邮件、营销推送、日志上报    │ 体验略差可补发   │ Best-effort ││
│  └──────────────────┴─────────────────────────────────┴──────────────────┴─────────────┘│
│  💡 EventBusCapability 统一入口，Host 层根据事件注解/配置决定是否走 Outbox              │
│  【来源】Netflix / Uber Outbox Pattern / CAP 定理最终一致性                              │
│                                                                                          │
└─────────────────────────────────────────────────────────────────────────────────────────┘
```

## 10. 关键设计原则

```text
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                              关键设计原则                                                │
├─────────────────────────────────────────────────────────────────────────────────────────┤
│                                                                                          │
│  原则 1：Capability Contract 优先                                                       │
│  ═══════════════════════════════════════════════════════════════════════════════════════│
│  • 插件只依赖 Capability 接口，永不依赖具体实现                                          │
│  • Capability 是稳定契约，实现可随 Host 变化                                             │
│  • 新 Capability 需经架构评审，避免接口泛滥                                              │
│                                                                                          │
│  原则 2：声明式配置优于命令式代码                                                        │
│  ═══════════════════════════════════════════════════════════════════════════════════════│
│  • 事件订阅通过 Manifest 声明，不写 @EventListener                                       │
│  • 插件组合通过 YAML 配置，不通过 pom.xml 硬编码                                         │
│  • 路由规则通过配置定义，不在代码中 if-else                                              │
│                                                                                          │
│  原则 3：基础设施隔离（Infrastructure Isolation）                                        │
│  ═══════════════════════════════════════════════════════════════════════════════════════│
│  • 插件代码中禁止出现 Kafka / Redis / HTTP Client 等依赖                                 │
│  • Runtime Shell 是唯一允许接触基础设施的层                                              │
│  • 通过编译时检查（ArchUnit）强制执行                                                    │
│                                                                                          │
│  原则 4：Host 极薄设计（Ultra-Thin Host）                                                │
│  ═══════════════════════════════════════════════════════════════════════════════════════│
│  • Host 是纯配置驱动的极薄组装壳，零实现代码、零业务逻辑、零工具类               │
│  • Host = pom.xml（依赖声明）+ YAML（配置）+ 启动类（< 30 行）                        │
│  • Standalone Host 与 Embedded Host 提供相同的 Capability 接口                          │
│  • 能力切换：更换 pom.xml adapter 依赖 + 修改 YAML，绝不改代码                    │
│  • 不为嵌入模式设计"特例架构"或"降级逻辑"                                                │
│                                                                                          │
│  原则 5：事件驱动解耦（Event-Driven Decoupling）                                         │
│  ═══════════════════════════════════════════════════════════════════════════════════════│
│  • 插件之间只通过 Integration Event 通信，无直接调用                                     │
│  • 事件路由由 Runtime Shell 管理，插件不感知目标                                         │
│  • 同步调用仅限于查询类能力（Query），修改类必须异步（Command）                           │
│                                                                                          │
│  原则 6：前后端完全分离（Full-Stack Separation）⭐ Updated                               │
│  ═══════════════════════════════════════════════════════════════════════════════════════│
│  • 前端采用 View → ViewModel → Repository 三层分离架构                                  │
│  • Repository 通过工厂模式注入 HttpCapability，封装全部后端 API 调用                     │
│  • {name}-shared 模块作为前后端类型契约（DTO、事件名、权限码）                           │
│  • UI 样式/交互大规模改动时，后端代码零改动                                              │
│  • 后端 API 变更时，仅需修改 Repository 层 + shared 类型                                 │
│  • 插件 UI 与宿主解耦，可嵌入任意 UI Host                                                │
│                                                                                          │
│  原则 7：可观测性内置（Observability by Design）                                         │
│  ═══════════════════════════════════════════════════════════════════════════════════════│
│  • 所有跨插件通信必须可追踪（Trace ID 透传）                                             │
│  • 事件流转全链路可视化                                                                  │
│  • Runtime Shell 负责埋点，插件无需关心                                                  │
│                                                                                          │
│  原则 8：渐进式演进（Evolutionary Architecture）                                         │
│  ═══════════════════════════════════════════════════════════════════════════════════════│
│  • 新增 Capability 不影响现有插件                                                        │
│  • Host 实现可独立升级                                                                   │
│  • 插件版本独立管理，支持灰度发布                                                        │
│                                                                                          │
└─────────────────────────────────────────────────────────────────────────────────────────┘
```

## 11. 与 v3.0 适配器方案对比

| 维度 | v3.0 Port/Adapter 方案 | v3.0 Runtime Shell 方案 |
|------|------------------------|-------------------------|
| **抽象层次** | 技术适配（Kafka → Webhook） | 能力契约（EventBus Capability） |
| **插件依赖** | 依赖 Port 接口 | 依赖 Capability Contract |
| **配置方式** | YAML 配置适配器类型 | YAML 配置 + Manifest 声明 |
| **事件模型** | 统一 Event | Domain Event + Integration Event 分层 |
| **组合方式** | pom.xml 依赖 | Solution YAML 声明式组合 |
| **前端架构** | 待定 | View → ViewModel → Repository 三层分离 + {name}-shared 类型契约 |
| **架构治理** | 代码规范约束 | Manifest 声明 + 编译时校验 |
| **商业支撑** | 适配器切换 | 完整售卖层次模型 |

**演进关系**：Runtime Shell 方案是 Port/Adapter 方案的**抽象提升**，而非替代。Port/Adapter 可作为 Runtime Shell 中 EventBusCapability 的具体实现策略。
